## 1. Executive Summary

**Vulture** is an AI-powered security audit assistant that enables developers and small teams to scan their GitHub repositories for vulnerabilities — without needing expensive penetration testing services. Users authenticate via GitHub (through Clerk), select a repository, and receive a prioritized vulnerability report generated by combining **Semgrep** static analysis, **AST/Tree-sitter** code parsing, and **AWS Bedrock Claude** AI reasoning.

### 1.1 Problem Statement

Security audits cost thousands of dollars. Small teams and indie developers ship vulnerable code because they can't afford professional pen-testing. Existing free tools (like `npm audit` or GitHub Dependabot) only catch dependency-level issues — they don't analyze application logic, design flaws, or OWASP Top 10 violations in the actual source code.

### 1.2 Solution

Vulture bridges this gap with a **three-layer analysis engine**: **Semgrep** for deterministic, rule-based vulnerability detection (OWASP Top 10, secrets, injection patterns), **AST/Tree-sitter parsing** for deep structural code understanding, and **AWS Bedrock Claude** for AI-powered contextual analysis. Together, they produce a comprehensive, Jira-style vulnerability report card with severity ratings, affected code snippets, CVE/CWE references, and actionable remediation steps.

### 1.3 Target Users

| Persona | Description |
|---------|------------|
| **Indie Developer** | Solo developer who wants a quick security check before shipping |
| **Startup Engineering Team** | Small team (2-10 devs) without a dedicated security engineer |
| **Open Source Maintainer** | Wants to ensure their public repo doesn't expose users to risk |
| **Student / Learner** | Wants to learn about security vulnerabilities in their own code |

## 2. Product Vision

> _"Scan a repo, get a security report card — 3 Critical, 7 High, 12 Medium — with code snippets and remediation steps."_

### 2.1 Core Value Proposition

- **Zero Setup:** Login with GitHub, pick a repo, get results. No CI/CD integration needed.
- **Deep Analysis:** Goes beyond dependency scanning — combines Semgrep rule-based detection with AST parsing for structural code analysis.
- **Hybrid Engine:** Deterministic Semgrep findings provide high-confidence base results; Claude AI enriches them with contextual analysis and catches patterns rules can't express.
- **AI-Powered Intelligence:** Uses Claude (via AWS Bedrock) to understand code context and identify nuanced vulnerabilities.
- **Actionable Reports:** Every finding includes severity, OWASP/CWE classification, affected code snippet, and a step-by-step fix.

### 3.3 Code Analysis Pipeline

| ID | Requirement | Priority |
|----|------------|----------|
| SCAN-01 | Clone the selected repository into an isolated temporary directory on the backend | P0 |
| SCAN-02 | Run **Semgrep** scan with OWASP Top 10 + security rulesets (`p/owasp-top-ten`, `p/security-audit`) and output results as JSON | P0 |
| SCAN-03 | Run **AST/Tree-sitter** parser across all supported source files | P0 |
| SCAN-04 | Extract structural code information: function signatures, imports, control flow, data flow, API endpoints, authentication patterns | P0 |
| SCAN-05 | Merge Semgrep findings + AST structure into a structured code summary suitable for LLM consumption | P0 |
| SCAN-06 | Support parsing for: Python, JavaScript/TypeScript, Java, Go, Ruby, PHP, C/C++ | P0 |
| SCAN-07 | Cleanup temporary directory after analysis is complete | P0 |
| SCAN-08 | Enforce maximum repo size limit (500 MB) to prevent abuse | P1 |
| SCAN-09 | Show real-time scan progress to the user (cloning → semgrep → parsing → analyzing → generating report) | P1 |

### 3.4 AI Vulnerability Analysis

| ID | Requirement | Priority |
|----|------------|----------|
| AI-01 | Send Semgrep findings + parsed code structure + raw code snippets to AWS Bedrock Claude for vulnerability analysis | P0 |
| AI-02 | Prompt Claude to identify vulnerabilities based on OWASP Top 10:2025 categories | P0 |
| AI-03 | Each vulnerability finding must include: title, severity (Critical/High/Medium/Low/Info), OWASP category, CWE ID, affected file + line numbers, code snippet, description, and remediation steps | P0 |
| AI-04 | Generate an overall security score (0-100) for the repository | P1 |
| AI-05 | Handle large codebases by chunking the analysis and merging results | P1 |
| AI-06 | Detect: hardcoded secrets, SQL injection, XSS, insecure deserialization, broken access control, SSRF, path traversal, command injection, insecure cryptography, outdated dependencies | P0 |
| AI-07 | Use Semgrep findings as high-confidence anchors; Claude validates, deduplicates, and enriches them with contextual reasoning | P0 |
| AI-08 | Tag each finding with its source: `semgrep` (rule-based), `ai` (Claude-detected), or `hybrid` (Semgrep-detected + AI-enriched) | P1 |

## 4. Non-Functional Requirements

### 4.1 Performance

| Requirement | Target |
|------------|--------|
| Repository clone time | < 30 seconds for repos under 100 MB |
| Semgrep scan time | < 45 seconds for repos with < 10,000 files |
| AST parsing time | < 60 seconds for repos with < 10,000 files |
| AI analysis response time | < 90 seconds per analysis chunk |
| Frontend initial load | < 2 seconds |
| API response time (non-scan) | < 500ms |

### 4.2 Security

| Requirement | Description |
|------------|-------------|
| Data isolation | Each scan runs in an isolated temp directory, cleaned up after use |
| Token security | GitHub tokens encrypted at rest, never exposed to frontend |
| API authentication | All backend endpoints protected with Clerk JWT verification |
| No code storage | Source code is never persisted; deleted after analysis |
| HTTPS only | All communications over TLS |

### 4.3 Scalability

| Requirement | Description |
|------------|-------------|
| Concurrent scans | Support at least 10 concurrent scan jobs |
| Queue management | Use background task processing for long-running scans |
| Rate limiting | Implement rate limiting per user (e.g., 10 scans/hour) |

### 4.4 Reliability

| Requirement | Description |
|------------|-------------|
| Error handling | Graceful degradation if AWS Bedrock is unavailable |
| Retry logic | Auto-retry failed Bedrock API calls (up to 3 attempts with exponential backoff) |
| Timeout management | Kill scan jobs that exceed 5-minute timeout |

## 5. OWASP Top 10:2025 Coverage

Vulture will detect vulnerabilities aligned with the latest OWASP Top 10:2025:

| # | Category | What Vulture Detects |
|---|----------|---------------------|
| A01 | **Broken Access Control** | Missing auth checks, IDOR patterns, privilege escalation, SSRF endpoints |
| A02 | **Security Misconfiguration** | Debug mode enabled, default credentials, overly permissive CORS, exposed admin routes |
| A03 | **Software Supply Chain Failures** | Known vulnerable dependencies, typo-squatting risk, unpinned versions |
| A04 | **Cryptographic Failures** | Weak hashing (MD5/SHA1 for passwords), hardcoded keys, missing encryption |
| A05 | **Injection** | SQL injection, NoSQL injection, OS command injection, LDAP injection, XSS |
| A06 | **Insecure Design** | Missing rate limiting, no input validation, race conditions, missing CSRF protection |
| A07 | **Authentication Failures** | Weak password policies, session fixation, missing MFA, JWT misconfigurations |
| A08 | **Software/Data Integrity Failures** | Insecure deserialization, unsigned updates, CI/CD pipeline risks |
| A09 | **Logging & Alerting Failures** | Missing audit logs, sensitive data in logs, no error monitoring |
| A10 | **Mishandling of Exceptional Conditions** | Catch-all exception handlers, failing open, information leakage in errors |

## 7. Data Models

### 7.1 User

```
User {
  id: UUID
  clerk_id: String
  github_username: String
  github_avatar_url: String
  github_access_token: String (encrypted)
  created_at: DateTime
  updated_at: DateTime
}
```

### 7.2 Scan

```
Scan {
  id: UUID
  user_id: UUID (FK → User)
  repo_full_name: String        // e.g., "octocat/hello-world"
  repo_url: String
  branch: String
  status: Enum(pending, cloning, semgrep_scanning, parsing, analyzing, completed, failed)
  security_score: Integer (0-100)
  summary: JSON {
    critical: Integer
    high: Integer
    medium: Integer
    low: Integer
    info: Integer
  }
  started_at: DateTime
  completed_at: DateTime
  created_at: DateTime
}
```

### 7.3 Vulnerability

```
Vulnerability {
  id: UUID
  scan_id: UUID (FK → Scan)
  title: String
  severity: Enum(critical, high, medium, low, info)
  detection_source: Enum(semgrep, ai, hybrid)  // how was this found
  semgrep_rule_id: String?      // e.g., "python.lang.security.audit.eval-detected" (null if AI-only)
  owasp_category: String        // e.g., "A01:2025"
  cwe_id: String                // e.g., "CWE-89"
  file_path: String
  line_start: Integer
  line_end: Integer
  code_snippet: Text
  description: Text
  remediation: Text
  confidence: Enum(high, medium, low)  // detection confidence level
  created_at: DateTime
}
```

## 10. Risks & Mitigations

| Risk | Impact | Mitigation |
|------|--------|-----------|
| AWS Bedrock rate limits / throttling | Scans fail or are delayed | Implement queuing, retry logic, and user-facing wait estimates |
| Large repos overwhelm parser | Timeout / OOM crashes | Enforce repo size limits; stream files instead of loading all into memory |
| Claude hallucinations (false positives) | Users lose trust | Cross-validate with static analysis rules; show confidence scores |
| GitHub token scope concerns | Users hesitant to grant access | Clearly document minimum required scopes; offer public-repo-only mode |
| Cost management (Bedrock per-token pricing) | Runaway costs | Implement per-user scan quotas; chunk efficiently to minimize token usage |

## 12. Glossary

| Term | Definition |
|------|-----------|
| **AST** | Abstract Syntax Tree — a tree representation of the syntactic structure of source code |
| **Tree-sitter** | An incremental parsing library that can build and update ASTs for source code |
| **OWASP** | Open Worldwide Application Security Project |
| **CWE** | Common Weakness Enumeration — a standardized list of software security weaknesses |
| **CVE** | Common Vulnerabilities and Exposures — a catalog of known security vulnerabilities |
| **SSRF** | Server-Side Request Forgery |
| **XSS** | Cross-Site Scripting |
| **IDOR** | Insecure Direct Object Reference |
